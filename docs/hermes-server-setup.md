# Hermes server setup (this laptop)

> **Audience:** Qodeca maintainer running Hermes as an always-on server on this Mac
> **Scope:** Qodeca-specific runtime setup, not part of upstream Hermes docs
> **Last updated:** 2026-07-08

Turns this MacBook into an always-on Hermes server that **runs from the fork source**
(`~/Projects/hermes-agent`, editable install, branch `main`). Surfaces: Telegram + Slack
gateway, a Tailscale-only web dashboard, and the local CLI. Model backend is the local
oMLX server (`127.0.0.1:8000`, `ornith-1.0-35b-a3b-4bit`), itself a launchd agent.

## Components

| Piece | Location |
|-------|----------|
| Dashboard launcher (crash-guard) | `~/.hermes/bin/hermes-dashboard-launcher.sh` |
| Health watchdog | `~/.hermes/bin/hermes-watchdog.sh` |
| Safe deploy (apply source edits) | `~/.hermes/bin/hermes-deploy.sh` |
| Dashboard service | `~/Library/LaunchAgents/ai.hermes.dashboard.plist` |
| Keep-awake service | `~/Library/LaunchAgents/ai.hermes.keepawake.plist` |
| Watchdog service | `~/Library/LaunchAgents/ai.hermes.watchdog.plist` |
| Gateway service | `~/Library/LaunchAgents/ai.hermes.gateway.plist` (generated by `hermes gateway start`) |
| oMLX model backend | `~/Library/LaunchAgents/com.local.omlx-serving.plist` (pre-existing) |
| Runtime config | `~/.hermes/config.yaml`, secrets in `~/.hermes/.env` |

All Hermes agents use `LimitLoadToSessionType = [Aqua, Background]` so they load in a
headless auto-login session after reboot.

## Runbook — remaining manual steps

### 1. Tailscale (needed for the dashboard bind)
```bash
brew install --cask tailscale     # pkg already downloaded; needs your sudo password
open -a Tailscale                 # sign in, approve the VPN system extension
tailscale ip -4                   # note the 100.x.y.z tailnet IP
```

### 2. Reboot resilience (System Settings, GUI)
- Privacy & Security → **FileVault → Turn Off** (wait for decryption to finish).
- Users & Groups → **Automatically log in as** your user.

### 3. Gateway (Telegram + Slack)
```bash
hermes gateway setup      # paste BotFather + Slack tokens (interactive)
hermes gateway start      # installs + bootstraps the launchd service
hermes gateway status
```

### 4. Dashboard password → I store the hash
Pick a strong password and tell me; I hash it with `plugins.dashboard_auth.basic.hash_password`
and write `dashboard.basic_auth.{username,password_hash,secret}` into `config.yaml`
(idempotent, mode 600). Or do it yourself:
```bash
PW="$(openssl rand -base64 24)"; echo "password: $PW"   # save it in your password manager
HASH="$(cd ~/Projects/hermes-agent && HP="$PW" .venv/bin/python -c 'import os;from plugins.dashboard_auth.basic import hash_password;print(hash_password(os.environ["HP"]))')"
# then add under a `dashboard:` block in ~/.hermes/config.yaml:
#   basic_auth:
#     username: admin
#     password_hash: "<HASH>"
#     secret: "<openssl rand -base64 32>"
```

### 5. Rotate the oMLX key (it appeared in a chat transcript)
Generate a new key, update `~/Projects/mlx/serving/secrets/`, and set `model.api_key` in
`~/.hermes/config.yaml`. The watchdog reads the key from config, so it keeps working.

### 6. Watchdog alerts (optional but recommended)
Add `WATCHDOG_TELEGRAM_CHAT_ID=<your chat id>` to `~/.hermes/.env`. `TELEGRAM_BOT_TOKEN`
is already set by step 3.

### 7. Load the dashboard (after 1 + 4 are done)
```bash
launchctl bootstrap gui/$(id -u) ~/Library/LaunchAgents/ai.hermes.dashboard.plist
tail -f ~/Library/Logs/hermes-dashboard.err.log   # confirm it stays up
```

## Day-to-day: applying your code changes

The server runs from the live working tree with **no auto-reload**. After editing:
```bash
~/.hermes/bin/hermes-deploy.sh    # smoke-imports, then restarts gateway + dashboard
```
It refuses to restart if the code doesn't import (broken edit / unresolved merge markers),
so a bad save can't crash-loop the services.

**Upstream merges:** stop the services first (`hermes gateway stop`), merge
`upstream/main`, resolve fully, run `hermes-deploy.sh`. Never merge into the live tree
while serving.

## Security posture

- Dashboard binds the **Tailscale IP only** (never `0.0.0.0`); reachable only on your tailnet.
- Basic-auth login; XFF is ignored unless `dashboard.trusted_proxy: true` (rate-limit fix).
- Treat the dashboard as a privileged admin console (it fronts a browser terminal + secret
  reveal). Use a long random password; don't expose beyond the tailnet without OAuth + TLS.
